# Layer 2.5 — Redundant Non-Colour Encoding (design)

**Status:** Design.
**Predecessors:** Layer 0 (`2026-06-07-colour-token-taxonomy-design.md`), Layer 1
(`2026-06-08-layer1-palette-sets-design.md`, `2026-06-09-layer1-highcontrast-design.md`),
Layer 2 (`2026-06-11-layer2-runtime-theme-design.md`).

## 1. Goal

The palette work in Layers 0–2 makes colours *configurable*, but five places in the editor
still encode meaning **only** via hue, which the Layer 1 palettes (including Colourblind and
High-Contrast) cannot fully fix on their own — meaning must survive when hue is
indistinguishable (~8% of men), or when the user can't distinguish colour at all. This layer
adds a second, non-colour cue (shape/icon/line-pattern/text-decoration) alongside each
existing colour, applied **across the whole solution**.

Research found the gap is narrower than the catalogue in `Gaps.md` implied: PatchManager
already pairs every `Brush.Severity.*` use with a ⚠ glyph, a tooltip, and/or adjacent text, so
**no changes are needed there**. All five areas below live in `DialogEditor.Avalonia`.

Non-goal: new colour tokens. Every change reuses existing `Brush.*` tokens — `NoStrayHexTests`
and `PaletteContrastTests` are unaffected.

## 2. Node-type shape markers

**Where:** `ConversationView.axaml`, the canvas node header's 6×6 "speaker-identity" `Ellipse`
(currently always an `Ellipse`, colour-coded via `SpeakerCategoryToBrushConverter` multi-binding,
`ToolTip.Tip="{Binding SpeakerCategory}"`).

**Change:** replace the `Ellipse` with a `Path` whose `Data` is selected by
`(SpeakerCategory, DisplayType)`, keeping the same fill (existing
`SpeakerCategoryToBrush`/Bark-aware multi-binding, unchanged) and the existing tooltip.
Enlarge from 6px to ~9px so the shape is legible.

| Type      | Shape    | Notes |
|-----------|----------|-------|
| Npc       | Circle   | matches "current default" baseline |
| Player    | Square   | |
| Narrator  | Triangle | |
| Script    | Diamond  | |
| Bark      | Star     | `DisplayType == "Bark"` overrides SpeakerCategory, mirroring `NodeColorConverter` |

**Implementation:** new converter `NodeTypeShapeConverter : IMultiValueConverter`
(`values[0] = SpeakerCategory`, `values[1] = DisplayType`), returns a `Geometry` on a 0–9
viewbox, parsed via `Geometry.Parse(...)`:

- Circle: `M4.5,0 A4.5,4.5 0 1 1 4.5,9 A4.5,4.5 0 1 1 4.5,0 Z`
- Square: `M0,0 L9,0 L9,9 L0,9 Z`
- Triangle: `M4.5,0 L9,9 L0,9 Z`
- Diamond: `M4.5,0 L9,4.5 L4.5,9 L0,4.5 Z`
- Star: approximate 5-point star within the 0–9 box, e.g.
  `M4.5,0 L5.6,3.3 L9,3.3 L6.2,5.4 L7.3,8.7 L4.5,6.6 L1.7,8.7 L2.8,5.4 L0,3.3 L3.4,3.3 Z`
  (exact coordinates may be refined for visual balance during implementation).

The same `Path`/converter pair is used in the `ConversationView` node header. **Minimap items
stay colour-only** — at minimap scale (~10×4px per node) a shape would be unreadable, and the
canvas remains the source of truth.

## 3. Diff status corner badges

**Where:** `ConversationView.axaml`, the diff-tint overlay `Border` (3px `BorderBrush` from
`DiffStatusToBrushConverter`, transparent for `Unchanged`).

**Change:** keep the existing coloured border unchanged. Add a small circular badge anchored
to the node's top-right corner (e.g. `Width=18 Height=18`, `CornerRadius=9`,
`HorizontalAlignment=Right VerticalAlignment=Top`, slight negative margin so it overlaps the
border), `Background` = the same `Brush.Diff.*.Fill` token (via the existing
`DiffStatusToBrushConverter`), containing a bold white `TextBlock`:

| DiffStatus | Glyph |
|------------|-------|
| Added      | `+`   |
| Changed    | `~`   |
| Removed    | `−`   |
| Unchanged  | (badge not rendered — `IsVisible` bound via a new bool converter, or glyph-converter returns `""` and badge collapses) |

**Implementation:** a new converter `DiffStatusToGlyphConverter : IValueConverter` returning
the glyph string (or `string.Empty` for `Unchanged`/`null`), paired with an
`IsVisible` binding using `StringIsEmpty`-style inversion (an existing converter pattern —
reuse rather than add a second new converter if a suitable one exists).

## 4. Connection line styles

**Where:** `ConversationView.axaml` `<UserControl.Styles>`, the `nodify|Connection.*` style
selectors.

**Change:** one-line addition —

```xml
<Style Selector="nodify|Connection.always">
    <Setter Property="Stroke" Value="{DynamicResource Brush.Connection.Always}"/>
    <Setter Property="StrokeDashArray" Value="10,2,2,2"/>
</Style>
```

`Default` stays plain solid (the "nothing special" baseline — no marks needed). `Never` keeps
its existing `4,3` dash (already non-colour-distinguishable from `Default`). `Highlighted*`
keeps its existing +1px thickness bump (already non-colour-distinguishable). During
implementation, verify `Connection.highlighted.always` still renders the dash-dot pattern
(Avalonia style/selector precedence — `.highlighted.always` must not silently drop the
`StrokeDashArray` set by `.always`); if it does, add an explicit `StrokeDashArray` setter to
the `.highlighted.always` selector too.

## 5. Severity icon (Flow Analytics)

**Where:** `FlowAnalyticsWindow.axaml`, the issue list's 4px severity bar
(`Background="{Binding Kind, Converter={StaticResource FlowIssueSeverityBrush}}"`), next to
the existing `KindLabel` text (already a distinct, readable per-row label — see
`FlowIssueViewModel.KindLabel`).

**Change:** widen the severity indicator from a 4px bar to a small (~16×16) rounded square
containing a glyph, same background colour as today:

| Severity (per `FlowIssueKindToSeverityBrushConverter`) | Glyph |
|---------------------------------------------------------|-------|
| Error (`Unreachable`)                                    | `⛔`  |
| Warning (everything else)                                | `⚠`  |

**Implementation:** new converter `FlowIssueKindToSeverityGlyphConverter : IValueConverter`,
sibling to `FlowIssueKindToSeverityBrushConverter`, same `FlowIssueKind` switch shape. Bind a
`TextBlock.Text` inside the existing `Border` (which keeps its `Background` binding).

## 6. Inline diff text decorations

**Where:** `DialogEditor.Avalonia/Controls/InlineDiffTextBlock.cs`, `MakeRun(...)` calls for
`DiffKind.MineOnly` (before-side, `Brush.Diff.Inline.Mine`, sky) and `DiffKind.TheirsOnly`
(after-side, `Brush.Diff.Inline.Theirs`, red). Used by `DiffWindow` and
`GitConflictResolutionWindow`.

**Change:** add `TextDecorations` alongside the existing `Foreground`:

- `MineOnly` (removed) → `TextDecorations.Strikethrough`
- `TheirsOnly` (added) → `TextDecorations.Underline`
- `Common` → unchanged (no decoration)

`MakeRun` gains an optional `TextDecorationCollection?` parameter (or two `MakeRun` overloads),
set on the returned `Run`.

## 7. Tests

- **`NodeTypeShapeConverterTests`** (new, `AvaloniaFact`/`AvaloniaTheory`-style like
  `BrushConverterTests`): one geometry per `(SpeakerCategory, DisplayType)` combination,
  asserting distinct `Geometry` per type and the Bark/star override regardless of
  `SpeakerCategory`.
- **`DiffStatusToGlyphConverterTests`**: `Added`→`+`, `Changed`→`~`, `Removed`→`−`,
  `Unchanged`/`null`→empty.
- **`FlowIssueKindToSeverityGlyphConverterTests`**: `Unreachable`→`⛔`, all other kinds→`⚠`
  (mirrors the existing brush converter's theory cases).
- **`InlineDiffTextBlockTests`** (extend existing coverage if present, else new): after
  `RenderDiff()`, `MineOnly` runs have `TextDecorations.Strikethrough` and `TheirsOnly` runs
  have `TextDecorations.Underline`, `Common` runs have none.
- Connection dash-array change: covered by an `AvaloniaFact` rendering a
  `Connection.always`/`Connection.highlighted.always`-classed control and asserting
  `StrokeDashArray`, if `ConversationView` styles are reachable in headless tests (otherwise a
  manual verification note in the implementation plan).

No changes to `NoStrayHexTests`, `PaletteContrastTests`, `PaletteSetParityTests`, or
`PaletteGoldenTests` — no new colours.

## 8. Out of scope

- Minimap node-type/diff shapes (colour-only retained — overview aid, not source of truth).
- PatchManager (`PatchManagerView.axaml`) — already pairs severity colours with ⚠ glyphs,
  tooltips, and/or adjacent text; no changes needed.
- Legend window — may warrant a follow-up update to document the new shapes/badges/icons, but
  is not required for the encoding itself to work. Tracked as a possible small addendum during
  implementation.
